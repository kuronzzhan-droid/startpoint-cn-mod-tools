## 1. 技能形态切换(スキルチェンジ)机制 —— 全在 ② 层 character 表

**切换条件不在 action_skill / DSL 里,而是 `master/character/character.orderedmap` 每行的 col9-16**
(CharacterValues.as 解析逆向,①层镜像 `assets/cdndata/character.json` 同列):

| 列 | 含义 |
|---|---|
| col8 | action_skill(基础技能键 = code_name) |
| col9 | **切换条件种类**:`(None)`=无切换;`0`=HpHigh;`1`=ConditionExist;`2`=MultiballNumber;`3`=ChangeSkillFlag;`4`=IsUnison |
| col10 | condition 状态枚举(kind=1 用,如 28) |
| col12 | multiballs 列表(kind=2 用,逗号分隔) |
| col13 | threshold(kind=0 的 HP 比例如 `0.5`;kind=2 的球数) |
| col14 | **切换后技能键**(指向 `master/skill/switched_action_skill.orderedmap`,按 code_name 索引,行含 program_path → 另一份 ActionDsl) |
| col15/16 | no_voice / no_ready_voice |

条件语义(SkillSwitchingConditionalKind.as,5 种):
- `0 HpHigh`:HP ≥ threshold(0-1 比例)时用 A 形态,低于则切 B
- `1 ConditionExist`:自身存在指定状态(col10 枚举)时切换
- `2 MultiballNumber`:场上多球数量 ≥ threshold 时切换
- `3 ChangeSkillFlag`:由词条/技能效果置"技能变化"标志触发(如用过一次后变形态)
- `4 IsUnison`:处于副位(unison)时切换

CN 库实例(共 6 个角色带切换,查 `cdndata/character.json` col9≠"(None)"):

| 角色 | kind | 参数 |
|---|---|---|
| 惠惠(megumin) | ConditionExist | cond=28, no_voice |
| 塔吉斯 / 贝瑞塔 / 贝尔赛蒂亚 | HpHigh | threshold=0.5 |
| 操里缘子(psychic_yamikawa) | MultiballNumber | multiball=1611291,1611292, threshold=1 |
| 莱特(light_chapter12) | ConditionExist | cond=3 |

**改法**:给任意角色加/改形态切换 = 改 character 表该行 col9-16(②层,发布生效);
切换后的技能本体在 `switched_action_skill` 表(键=code_name)+ 其 program_path 的 DSL。
`wf_publish.py` 需补 `switched_action_skill` 别名(逻辑路径 `master/skill/switched_action_skill.orderedmap`)。

## 2. 全资产包(datamine dump)与 store 的对应关系 —— 可一比一批量导入

pirates_girl.rar 实测(408 文件):**305 个文件与 store 逻辑路径
`character/<code>/<相对路径>` 一一对应**(wf_assets.locate 全命中),其余 84 个是
提取器自产物(sprite sheet 逐帧切片、GIF、`*_resized.png`、解码后的 `*.frame/timeline/atlas.json`),
游戏内不存在对应文件,导入时跳过即可。

当前 GUI 资产页未覆盖、但 store 确认存在的独立文件(全部 medium 根):

| 类别 | 逻辑路径(character/<code>/ 下) |
|---|---|
| 方形头像 4 组 | `ui/square_{0,1}.png`、`ui/square_132_132_{0,1}.png`、`ui/square_round_95_95_{0,1}.png`、`ui/square_round_136_136_{0,1}.png` |
| 缩略图 3 组 | `ui/thumb_level_up_{0,1}.png`、`ui/thumb_party_main_{0,1}.png`、`ui/thumb_party_unison_{0,1}.png` |
| 战斗 UI | `ui/battle_control_board_{0,1}.png`、`ui/battle_member_status_{0,1}.png` |
| 连锁 cut-in | `ui/cutin_skill_chain_{0,1}.png` |
| 剧情横幅 | `ui/episode_banner_0.png` |
| 剧情表情差分 | `ui/story/*.png`(base_0/angry/joy/sad/... 每角色约 18 张,独立文件非图集) |
| 剧情语音 | `voice/words/*.mp3`(pirates_girl 有 241 条;此前误记"仅剧情 NPC 用",实为角色也有) |

→ 结论:**"资产包导入"功能可行且简单**:选目录 + code_name → walk 相对路径拼逻辑路径 →
locate 命中的批量替换(自动 medium/upload 分根、混淆编码、备份、进 pending),未命中列清单跳过。
注意 `_0/_1` = 进化前后两套;图集类(illustration_setting/pixelart)仍须同尺寸同布局。

## 3. 角色资料页 稀有度/元素 同步现状(缺口,待修)

`save_char_fields` 目前**只写 ①层** `assets/cdndata/character.json` / `character_text.json`。
但 rarity/element 有三个消费方:

| 消费方 | 数据源 | 现状 |
|---|---|---|
| 游戏内显示/战斗(星级、属性图标、克制) | ②层 `master/character/character.orderedmap` col2/col3 | **改不到,游戏内无变化** |
| 服务端逻辑(邮件发放校验、admin) | `assets/character.json`(简化表 {name,rarity,element,skill_count}) | **改不到** |
| GUI 自身显示 | ①层 cdndata JSON | ✅ 唯一生效处 |

→ 修复方向:save_char_fields 对 master 字段(rarity/element/code_name/race/gender)
同步写 ②层 character 表同名列 + `assets/character.json`,②层进 pending 走发布。
文本字段(名字/描述等)维持 ①层即可?——**否**:客户端名字/描述实际读 ②层
`character_text.orderedmap`,①层 cdndata 只是镜像。文本字段同理需要同步 ②层才在游戏内生效
(待验证:先金丝雀改一个名字发布看效果,再定同步范围)。
