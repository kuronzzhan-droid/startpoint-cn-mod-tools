# WF 修改器 · 版本切换重新设计

> 目标:把"当前在改哪个版本(CN / global / 未来新版)"变成**显式、成套、可切换**的概念,
> 根除词条数值"引用国际服"的问题。
> 本文所有结论均在工程目录实测(CN 包 `弹国服/`、global 包 `wf-assets/`)核对确认。

---

## 一、问题定位:global 到底从哪里泄进来

先厘清一个易误判点 —— **文本词条(名字 / 技能描述)已经是 CN 的**
(`character_text.json` 里是"白""瓦格纳")。真正引用国际服的是**词条数值结构(ability schema)**。

实测对比两个数据包的 ability schema:

| | 列数 | 0–80 列 | 81 列起 |
|---|---|---|---|
| **CN(弹国服)** | **125** | 与 global 相同 | CN 多插 `powerflip_override.*` 等字段,整体后移 |
| **global(wf-assets)** | **119** | — | 布局不同 |

关键错位实证(同一逻辑字段,两版下标不同):

```
trigger.values.during_content.values.strength.power1   ← “持续威力”
   GLOBAL 下标 = 109
   CN     下标 = 112      ← 差 3 位
```

### 三处 global 泄漏点(按危害排序)

**1. 硬编码数字下标(核心问题)** —— 数字是从 global 119 列数出来的,用在 CN 上指向错列:

| 位置 | 硬编码 | global 含义 | 用在 CN 上实际指向 |
|---|---|---|---|
| `wf_gui.py:220` `describe_ability` | `108/109` | 持续威力 A/B | `during_content`(结构头)/ `target` ❌ |
| `wf_gui.html:95` 提示文案 | `108/109` | 效果 A/B | 同上 ❌ |
| `wf_mod_tool.py:650` `cmd_list` 默认列 | `108,112,114,122,123` | 持续 / 开幕数值 | 全部错位 ❌ |

> `0–80` 列两版一致,所以**瞬发**类(`50/52/54/56/58`)显示正确;**持续 / 开幕**类(>80)在 CN 下全错。

**2. global 数据包作为静默兜底** —— `wf_mod_tool.py:267`:

```python
def default_source_store():
    candidate = Path(r"D:\WF\wf-assets\upload")   # ← global 包
```

`load_table` 逻辑是"CN 读不到就读 source(global)"。CN 包目前完整所以没触发,
但**一旦某张 CN 表是 stub / 缺失,会静默读 global 数据**——你以为在改 CN,其实改了国际服。

**3. 硬编码 schema hash** —— `wf_mod_tool.py:44` `2b/6ca08e...` 恰好版本安全
(同逻辑路径 → 同 hash,内容各包各自),但它是裸魔法值,换版本若逻辑路径变了会静默失效。

### 有一个设计是对的(要保留)

`ABILITY_ALIASES` + `expand_fields` 走**按列名解析**(`schema_index`),名字→下标由当前
加载的 schema 决定。实测 `during_content.values.strength.power1` 在 CN 下能正确解析到 112。
**所以 scale / 逐字段编辑的写入是版本安全的**,坏的只是绕过 schema、直接写死数字的启发式显示和默认列。

---

## 二、根因

**工具里没有"当前是哪个版本"这个概念。** 它把三样东西隐式混在一起:
目标数据包(CN)、兜底源(global)、写死的列号(global)。没有一个统一的"版本"把它们绑成一致的一组。

---

## 三、核对数据:名录源是否与手机包同版本

GUI 左侧角色名录读的是 `assets/cdndata/character.json`(服务端资产),
需确认它与 `弹国服/` 手机包是否同一版本。实测:

### 3.1 cdndata 名录 ↔ CN 手机包:完全一致(同一版本)

| 检查项 | 结果 |
|---|---|
| 角色数 | 505 = 505 ✅ |
| 角色 ID 交集 | 505 / 505,零差异 ✅ |
| **ability 引用**(每角色 19–24 列共 2934 个 ability ID) | 在 CN store 中**全部存在,0 缺失** ✅ |
| code_name(列0) | 0 不一致 ✅ |
| 整行 37 字段逐列对比 | **0 差异** ✅ |

结论:`cdndata/character.json` 就是 CN store `character.orderedmap` 的 JSON 序列化副本,字段级完全吻合。

### 3.2 cdndata ↔ global 包:明显不同(证明 cdndata 属 CN)

| 检查项 | 结果 |
|---|---|
| global 角色数 | **475**(比 CN 少 30) |
| cdndata 独有角色 | **30 个 global 没有** |
| 共有角色字段差异 | 455 / 475 行不一致 |

30 个 CN 独有角色 + 大量字段差异,坐实 **cdndata 是 CN 版**,不是 global。

### 3.3 对设计的影响

- **好消息**:名录层当前与 CN store 天然对齐,无实际错位 —— "引用国际服"只在
  **ability schema 数字下标**和 **global 兜底源**两处;名录是干净的。
- **但仍要把 cdndata 纳入 profile 绑定**:现在对齐是"凑巧两边都是 CN"。一旦切到 global 档案
  (475 角色),cdndata 若还指向 CN 的 505,GUI 左侧列表和实际 store 就会错位
  (30 个角色点开找不到词条)。**每个 profile 必须 store + cdndata + schema 三者同版本一起切。**

## 四、重新设计:引入「版本档案 Version Profile」

核心思想:**一次只激活一个版本档案,所有版本相关的东西都从档案派生,禁止跨版本兜底。**

### 4.1 Profile 数据结构

```python
@dataclass
class VersionProfile:
    id: str                 # "cn" | "global" | "cn-1.4.54"
    label: str              # "国服(雷霆)"
    store: Path             # 该版本 upload 数据包(唯一数据源)
    cdndata: Path           # 配套名录(与 store 同版本)★必须绑定
    res_version: str = ""   # "1.4.54",用于校验 / 显示
    fallback: str | None = None   # 只允许指向"同版本更全包",默认 None
    # schema 不写死:按逻辑路径 SHA1 现算 → 从本档案 store 读
    # 列号不写死:从本档案 schema 派生 name→index 映射
```

配置文件 `mod-tools/profiles.json`:

```json
{
  "active": "cn",
  "profiles": {
    "cn": {
      "label": "国服(雷霆)",
      "store": "弹国服/WorldFlipper/dummy/download/production/upload",
      "cdndata": "assets/cdndata",
      "res_version": "1.4.54"
    },
    "global": {
      "label": "国际服(参考)",
      "store": "../wf-assets/upload",
      "cdndata": "assets/cdndata-global",
      "res_version": ""
    }
  }
}
```

### 4.2 五条原则

1. **单一数据源**:CN 档案只读 CN store。global 作为**独立档案**存在,绝不做 CN 的兜底。
   缺表就报错,不静默替换。
2. **列号全部运行时派生**:删掉所有裸数字下标,统一用 `col("during_content.values.strength.power1")`
   按名字取。后端把 `name→index` 映射下发前端,HTML 里不再出现 `108/109`。
3. **schema 按逻辑路径现算**:`table_path(store, ABILITY_SCHEMA_LOGICAL)` 取代硬编码 hash。
4. **版本指纹检测**:用 schema 列数(125=CN / 119=global)或包内 `info.json` 的 resourceVersion
   自动识别,UI 顶栏显示"当前:国服 1.4.54",加载到不匹配的包时告警。
5. **可切换 + 可持久化**:GUI 顶栏加下拉切版本;命令行 `--profile cn`;配置存 `profiles.json`。

### 4.3 落地改造点(增量,不推翻现有引擎)

| 步骤 | 改动 | 影响面 | 状态 |
|---|---|---|---|
| **P0** | 新增 `profiles.json` + `resolve_profile()`;`TARGET/SOURCE_STORE` 从 profile 取 | 小,纯新增 | ✅ 已完成 |
| **P1** | `load_ability_schema` 改为按 profile.store 读、按逻辑路径算 hash;删跨版本兜底 | 中 | ⏸ 暂缓 |
| **P2** | 前端 `describe_ability` / HTML 改成查名字表,**删除所有写死下标**(核心修复) | 中 | ✅ 已完成 |
| **P3** | `cmd_list` 默认列改为按别名(`skill_strength` 等)而非裸数字 | 小 | ✅ 已完成 |
| **P4** | GUI 顶栏加版本下拉 + 指纹校验告警 | 小 | ⏸ 暂缓 |

> **范围决策(2026-07-05)**:当前以 **CN 服功能为主**,版本切换功能暂不实现。
> - 已完成 **P0 + P2 + P3**:profile 抽象就位、默认锁定 CN、词条数值不再引用国际服。
> - **P1 暂缓**:`ABILITY_SCHEMA_REL` 硬编码 hash 目前版本安全(同逻辑路径→同 hash,各包读各自内容),CN 单版本下无需改。
> - **P4 暂缓**:GUI 顶栏仅**显示**当前版本(`国服 1.4.54` badge),不做下拉**切换**。
> - profile 结构已支持多版本,将来要恢复切换,补 P1/P4 即可,无需回改现有代码。

### 4.4 为什么这样切干净

- 现有 orderedmap 引擎、AMF3 解析、别名系统**完全复用** —— 它们本就名字驱动、版本无关。
- 真正要做的两件事:**(a) 把隐式的三个源收拢成显式 profile**;
  **(b) 把散落的写死下标改成按名字派生**。
- 未来出新版本(CN 2.x / 日服),只需放新包 + 加一条 profile,不用改代码。
