# 角色生成器使用说明

> 适用方案：角色生成器与编辑器 v2。
>
> 本手册面向“从中文概念生成一个可预览、可组包的新角色”的使用者。生成器负责生成、编辑、
> 校验和组包；新角色发布始终交给 `wf_character_flow.py`。完整设计依据见
> [角色生成器方案](角色生成器方案.md)，发布门禁依据见[新角色包工作流](角色包工作流.md)。

## 1. 先记住四条边界

1. 生成阶段只写 `work/char_gen/<code_name>/`；组包阶段只写
   `work/character_packs/<package_id>/package/`。
2. 生成器不得直接写 live store、`assets/cdndata`、`.cdn`，不得直接调用 `wf_publish.py`
   发布整角色包。
3. 一键生成的终点是“组包、`status`、`preflight`”；`runtime_test`、production 发布和回滚
   都必须由用户显式确认。
4. `story`、`words`、`login`、`episode`、剧情表情和剧情横幅不在生成范围，也不计入
   37 项发布门槛；不要为了凑数复制或占位。

如果网页里没有“角色生成”页签，或项目中没有 `mod-tools/wf_char_gen.py`，说明当前 checkout
尚未合入完整生成器。此时可以阅读和准备素材，但不要把下文命令当成已经可用的入口。

## 2. 推荐路线

第一次制作时使用：

- 数据模式：`assemble`（官方组装）；
- 流水线：`draft`（初版）；
- 像素模式：`recolor`（调色）；
- 语音：draft 阶段跳过；
- 人工停点：锁定母版、确认数据构建单、确认发布。

这条路线先用最低成本得到可在真机观察的角色。外观和方向确认后，再在同一个 `spec.json`
上升级为 `complete`。不要一开始就做 40–80 帧逐帧重绘和全量语音。

## 3. 环境准备

### 3.1 基础环境

在项目根目录 `D:\WF\startpoint-cn` 操作。至少需要：

- 项目依赖已按仓库要求安装；
- Python 能运行 `mod-tools/` 下脚本；
- ffmpeg 可由 `WF_FFMPEG` 或 `PATH` 找到；
- 使用 OpenAI 文本、图像或后备语音时，已配置 `OPENAI_API_KEY`；
- 使用本地语音时，GPT-SoVITS `api_v2` 已在本机启动。

启动网页修改器：

```powershell
python mod-tools/wf_gui.py
```

浏览器通常打开 `http://127.0.0.1:8765`。端口被占用时工具会选择备用端口，以终端输出为准。
进入“角色”模式，再打开“角色生成”。

### 3.2 配置 OpenAI API Key

本机已在用户级环境变量配置 `OPENAI_API_KEY` 时，不要把 Key 再写进 `spec.json`、提示词、
状态文件或聊天记录。只检查“是否存在”，不要打印值：

```powershell
$configured = -not [string]::IsNullOrWhiteSpace(
  [Environment]::GetEnvironmentVariable('OPENAI_API_KEY', 'User')
)
if ($configured) { 'OPENAI_API_KEY: configured' } else { 'OPENAI_API_KEY: missing' }
```

需要新配 Key 时，优先在 Windows“编辑账户的环境变量”中新增：

- 变量名：`OPENAI_API_KEY`
- 变量值：自己的 API Key

可选的代理或中转地址使用 `OPENAI_BASE_URL`，必须包含与 OpenAI API 兼容的 `/v1` 根地址。
修改环境变量后重新打开终端和网页修改器。生成器的查找顺序为：当前进程环境变量 → Windows
用户环境变量 → gitignored 的本地配置。除非确有隔离需要，不建议把 Key 放进配置文件。

Key 缺失时，AI 生成按钮应置灰并显示配置指引；手动上传、编辑已有产物、QA 和组包仍可使用。

### 3.3 配置 GPT-SoVITS 本地语音

GPT-SoVITS 是 v2 的默认语音 provider，适合中日双语和经授权的少样本声线。Windows 用户可用
[官方仓库](https://github.com/RVC-Boss/GPT-SoVITS)发布的整合包；也可按官方说明创建
Python 3.10 环境并运行安装脚本。不要混用来路不明的权重或参考音频。

源码安装的典型流程：

```powershell
git clone https://github.com/RVC-Boss/GPT-SoVITS.git
Set-Location GPT-SoVITS
conda create -n GPTSoVits python=3.10
conda activate GPTSoVits
pwsh -F install.ps1 --Device CU128 --Source HF
```

`--Device` 和 `--Source` 按本机显卡与下载源选择；CPU 环境使用 `CPU`。整合包用户按官方说明
先在 WebUI 中选择或训练 GPT/SoVITS 权重，并确认推理 WebUI 能正常合成，再启动 API v2：

```powershell
python api_v2.py -a 127.0.0.1 -p 9880 -c GPT_SoVITS/configs/tts_infer.yaml
```

端口检查：

```powershell
Test-NetConnection 127.0.0.1 -Port 9880
```

看到 `TcpTestSucceeded : True` 只证明端口在线；还要用相同权重、参考音频和参考文本做一条实际
合成。`api_v2` 的 `/tts` 请求至少需要正文、正文语言、参考音频路径、参考文本和参考语言。

生成器的本地 provider 配置目标如下；配置中不存 API Key：

```json
{
  "tts": {
    "provider": "local_http",
    "base_url": "http://127.0.0.1:9880",
    "defaults": {
      "text_lang": "ja",
      "prompt_lang": "ja"
    }
  }
}
```

每个角色还要填写固定声线卡：

```json
{
  "provider": "local_http",
  "ref_audio_path": "D:/voices/authorized/reference.wav",
  "prompt_text": "与参考音频完全一致的文本",
  "prompt_lang": "ja",
  "source_license": "用户自录并授权用于本项目"
}
```

同一角色的全部台词必须使用同一张声线卡。`source_license` 为空时 voice 阶段应拒绝运行。
只允许使用用户自录、明确授权、开源许可或 AI 生成的种子声线；禁止克隆真人声优或未授权音源。

如果界面未列出 `local_http`，或提示该 provider 尚未接入，说明当前版本只保留了接口；这不是
端口或配置错误。此时改用 OpenAI 后备，不要修改 live 数据绕过生成器。

### 3.4 OpenAI 语音后备

本地引擎不可用时，把声线卡 provider 改为 `openai`：

```json
{
  "provider": "openai",
  "model": "gpt-4o-mini-tts",
  "voice": "alloy",
  "instructions": "年轻、克制、清晰的日语动漫角色声线；语速稍慢",
  "speed": 0.95,
  "source_license": "OpenAI 生成声线，不模仿真人"
}
```

项目 v2 的后备模型名以当前配置为准。模型可用性和生命周期会变化；如果模型被停用，应先核对
当前 OpenAI 官方文档和项目兼容性，不要只改模型字符串后跳过试听。OpenAI 后备的日语动漫感
通常弱于合适的本地声线，必须人工试听。

无论使用哪个 provider，输出链路都是：WAV → ffmpeg 转 44.1 kHz、单声道、96 kbps CBR、
`-write_xing 0` → `wf_assets.mp3_encode` 严格预检。VBR、损坏文件或半截转换文件不能组包。

## 4. 成本预估与控制

以下为 2026-07-17 的美元估算，不含税、汇率、本地电费和重试。价格会变化，执行前以
[GPT Image 1 官方模型页](https://developers.openai.com/api/docs/models/gpt-image-1)、
[GPT-4.1 mini 官方模型页](https://developers.openai.com/api/docs/models/gpt-4.1-mini)和
[GPT-4o mini TTS 官方模型页](https://developers.openai.com/api/docs/models/gpt-4o-mini-tts)
为准。

| 项目 | 当前计价基准 | 本流程中的估算 |
|---|---:|---:|
| `gpt-image-1` 1024×1536，高质量 | US$0.25/张输出 | draft 一张基础母版约 US$0.25；complete 两张母版合计约 US$0.50 |
| `gpt-image-1` 1024×1024，中质量 | US$0.042/张输出 | 40–80 帧 repaint 的输出侧约 US$1.68–3.36 |
| `gpt-image-1` 1024×1024，高质量 | US$0.167/张输出 | 40–80 帧 repaint 的输出侧约 US$6.68–13.36 |
| `gpt-4.1-mini` 文本 | 输入 US$0.40/百万 token；输出 US$1.60/百万 token | 一次身份文本和构建建议通常远低于图像成本，以实际 token 为准 |
| `gpt-4o-mini-tts` | 文本输入 US$0.60/百万 token；音频输出 US$12/百万 audio token | 按最终音频 token 结算，生成后看用量；不要用“台词条数”冒充精确金额 |
| GPT-SoVITS 本地 | 无 API 调用费 | 只计本机算力、下载和维护成本 |

图像编辑还可能收取参考图输入 token；表中逐帧费用只是输出侧下限。特效重皮会额外增加帧数，
失败重试和“重新生成”会线性增加成本。开始 repaint 前，页面应显示实际帧数和预计调用数。

控制成本的顺序：

1. 先用 `draft + assemble + recolor`；
2. 锁定基础母版后再派生 UI，不要靠反复生成小图修构图；
3. 整体风格确认后才生成进化母版；
4. 像素 repaint 只在调色确实不能满足需求时启用；
5. 优先使用本地 GPT-SoVITS，OpenAI TTS 作为后备；
6. 不要随意使用 `--force`，输入和哈希未变化时让缓存命中。

## 5. 三种数据模式

| 模式 | 适合场景 | 数据来源 | 风险与门禁 |
|---|---|---|---|
| `assemble` | 第一次制作、求稳、快速真机 | 官方技能、词条、队长技、PF 重新组合 | 默认推荐；数值沿用官方先例 |
| `create` | 确实需要原创词条、技能或特效重皮 | LLM 起草，命令库/枚举组装，新 DSL/新路径 | 无先例组合；必须先 draft runtime_test 真机 |
| `hybrid` | 大框架求稳，只在少数槽位原创 | 每个槽位单独声明 official/new/composed/reskin | 只要含任一 `new` 槽位，就按 create 风险处理 |

`assemble` 不是简单复制整名角色：技能、每条词条、队长技和 PF 都可以分别选官方候选，工具会
适配元素、sid 和觉醒门槛。`create` 也不是让 AI 自由写二进制；必须经过签名、枚举、空值哨兵、
`custom_ability_string` 等合法性门禁和人工确认。

v2 不自动做数值平衡。原创结构的数值默认取官方先例；后续平衡调整仍在现有“词条编辑”、
“技能·倍率”和“效果构建器”中完成。

## 6. draft 与 complete 两段式

### 6.1 draft 初版

draft 的目标是“尽快得到可观察的角色”，范围为：

- 生成全套身份文本；
- 只生成基础母版，进化位暂时复用基础母版；
- 确定性派生全部必需 UI；
- 像素图使用 `recolor`；
- 跳过语音；
- 数据默认保留模板技能/词条，只换身份文本，或只应用已确认的最小构建单；
- 跑 draft QA、组包、37/37 检查和 preflight。

draft 的真机通道是 `runtime_test`，不是 production。每次都需要用户显式授权和精确口令
`DIRECT_REAL_TEST`；生成器不得代填授权字段。

### 6.2 complete 完整版

在同一个 `spec.json` 上升级：

1. 生成真正的进化母版并重派生进化侧 UI；
2. 选择是否把像素 `recolor` 升为 `repaint`；
3. 按模板实际 voice 清单生成 ally/battle/home 全量语音；
4. 完成数据构建单全部槽位；
5. 全量 QA 通过后重新组包；
6. live 漂移时 `rebase → preflight`；
7. 停服后走 production 发布。

runtime_test 包不能静默升级成 production。complete 必须重新走生产门禁，并绑定正确的已安装包。

## 7. 新建与一键生成

### 7.1 网页向导

在“角色生成”页选择“新建角色”，依次填写：

- 中文概念：外观、身份、性格、属性、武器和战斗主题；
- 可选参考图：只能作为风格或角色锚，立绘输出仍须透明背景；
- 模板角色：优先选同属性、资产清单完整、动作气质接近的 ★5；
- 数据模式：首次选 `assemble`；
- tier：首次选 `draft`；
- 新角色 ID：接受空闲 ID 建议或手工覆盖；
- `code_name`：小写英文、数字和下划线，不能与现有角色或 store 路径冲突；
- `package_id`：必须先看第 11 节的单包所有权约束，不能自行起独立生产包名。

“一键生成”默认 `auto=false`，会在三个位置停下：

1. 基础母版锁定；
2. 数据构建单确认；
3. 发布授权。

`auto=true` 只会自动接受第一张母版和 LLM 首选构建单，仍必须在发布点停下。初次制作不建议
使用 `auto=true`。

### 7.2 CLI 一键初版

```powershell
python mod-tools/wf_char_gen.py oneclick `
  --concept "暗属性重甲龙骑士，沉默寡言，以护盾和反击保护队伍" `
  --mode assemble `
  --tier draft `
  --template 111165
```

可选参考图使用 `--ref <PNG路径>`；明确需要自动接受前两个停点时才加 `--auto`。命令终点仍然
只是 pack、status 和 preflight，绝不自动 publish。

### 7.3 断点续跑

生成工作区固定为：

```text
work/char_gen/<code_name>/
├─ spec.json
├─ authoring/
│  └─ history/
├─ build/
├─ qa/
└─ cache/
```

`spec.json` 是唯一事实源。重新打开同一角色时从它恢复，不要从 `build/` 反推规格。每个阶段会
比较输入、缓存和产物哈希；未变化的完成项应跳过。只有确认需要无条件重算时才使用 `--force`。

逐阶段 CLI 的通用形式：

```powershell
$spec = 'work/char_gen/<code_name>/spec.json'
python mod-tools/wf_char_gen.py <stage> --spec $spec
```

其中 `<stage>` 为 `plan`、`text`、`masters`、`ui`、`pixel`、`voice`、`data`、`qa` 或
`pack`。实际可用参数以 `python mod-tools/wf_char_gen.py --help` 为准。

## 8. 逐阶段操作

### S0 plan：模板、身份与 workspace

输入：概念、参考图、模板 ID、目标属性、数据模式、tier。

操作：

1. 检查模板资产清单和实际尺寸；尺寸以 store 实测为准，不使用硬编码表。
2. 扫描角色表，分配空闲角色 ID。
3. 校验 16 表无键冲突、`code_name` 未被使用、`character/<code_name>/` 不存在。
4. 按第 11 节确定 `package_id`，初始化对应 flow workspace。
5. 检查 spec 中模板 ID、新角色 ID、code、package ID 四者一致。

通过标准：冲突列表为空；workspace 已建立；没有写 live store、assets 或 CDN。

### S1 text：身份文本与台词稿

输入：概念、模板真实文本、身份卡。

检查并编辑：

- 中英文名、称号、描述；
- 技能名、技能描述及进化版；
- 队长技称号；
- CV 显示名，应标注“AI 声线”，不能冒用真人声优；
- 发色、瞳色、服装、配色等身份不变量；
- 按模板实际 voice 清单一一对应的中文稿和日语台词。

模板有多少条语音路径就生成多少条稿，不多不少。台词修改后，对应语音会在调整循环中变
`dirty`；文本表声明也会更新。不要生成 story/words/login 文本或资产。

### S2 masters：基础与进化母版

输入：概念、身份卡、可选参考图。

要求：

- 透明背景、完整全身、裁剪边界清晰；
- 基础与进化必须是同一人物；
- 不能把带场景背景的图直接当立绘母版；
- draft 只需基础母版，进化位标记 `reuse_base`；
- complete 需要独立进化母版。

每张图都可重新生成或手动上传。先检查轮廓、服装不变量、手脚完整和透明背景，再点击“锁定”。
母版未锁定时不得进入 UI 派生。

### S3 ui：确定性派生 UI

此阶段不再调用 AI，而是从锁定母版按模板实际尺寸派生：

- 基础/进化 full shot；
- 头像、缩略图、战斗 UI、连锁 cut-in；
- `illustration_setting_sprite_sheet` 各 cell；
- `skill_cutin_0/1.png` 及 android ATF；
- 立绘定位和 `character_image`、`full_shot_image_attribute`、`trimmed_image` 三表声明。

查看 `qa/` 中的 UI 马赛克，重点看脸部是否在小图中心、武器是否误切、进化版是否串图、立绘
内容框是否越界。发现问题优先调整母版或裁切焦点，不要手工改 25 张派生图制造不可复现状态。

### S4 pixel：像素动画

`recolor` 模式：

- 查看“模板色 → 新色”映射；
- 修正肤色、头发、服装主色和高光；
- atlas、frame、timeline 原样不动；
- 生成后逐序列播放 GIF。

`repaint` 模式：

1. 确认页面显示的实际帧数和预计调用数；
2. 每帧保持原姿势、剪影、透明背景和 cell 边界；
3. 结果缩回原尺寸并量化到不超过 32 色；
4. alpha bbox IoU 必须 ≥ 0.6；
5. 任一漂移帧单独重生成，不能静默通过；
6. 全帧通过后才按原 atlas 坐标重组 sheet。

模式 B/C 的特效重皮使用同一规则，但只重绘贴图；atlas、`.parts` 骨架和时间线必须保持原样。
v2 不支持新 flatomo 骨架、新时间线或新 SFX。

### S5 voice：语音

draft 跳过。complete 对模板实际存在的 ally/battle/home 清单逐条执行：

1. 检查声线卡和 `source_license`；
2. 合成 WAV 到 `authoring/`；
3. ffmpeg 转 CBR MP3 到 `build/`；
4. 运行 MP3 严格预检；
5. 逐条试听，修改文本、语速或语气后只重生成该条。

至少抽听 join、battle skill 和 home 各一条代表样本。听感通过不等于编码通过，编码通过也不等于
角色一致性通过；两项都要验收。

### S5.5 data：数据构建单

`assemble`：从全库候选中为技能、每条词条、队长技和 PF 各查看出处、中文摘要和推荐理由，
人工勾选。适配后的表行应保留官方数值，并自动处理元素、sid、觉醒门槛。

`create`：逐槽位检查 LLM 起草内容、命令来源、参数签名、枚举、空值哨兵和先例数值。原创技能
必须使用新逻辑路径；InvokeSkill 必须注册 `custom_ability_string`。合法性校验失败时退回修改，
不能手改 SHA 或跳过门禁。

`hybrid`：逐槽位确认 `official`、`new`、`composed`、`reskin` 来源。任何 `new` 槽位都会把
整包提升为原创风险等级。

如果目标属性与模板不同，必须在组包表声明中完成整体转属性：角色 c3、词条/队长技属性 token、
content 元素列一起变。不能只改资料页显示。

数据产物只落 `build/data/`，此阶段不得调用 composer 直接写 live。

### S6 qa：汇总校验

必须检查：

- 必需资产差集为空；
- PNG 可解码且尺寸与模板同名资产一致；
- UI 马赛克、像素 GIF、可选特效预览通过；
- repaint 的 IoU 门全绿；
- MP3 全过严格校验；
- 数据行全过合法性哨兵；
- 三层声明一致；
- 报告没有未解释的 error。

draft 只允许两项明确豁免：语音缺席、进化母版复用基础母版。豁免必须写入报告，不能表现为
“假绿色”。产物为 `qa/report.json` 和页面可读报告；报告不绿时 pack 按钮不应可用。

### S7 pack：交给角色包工作流

pack 会把 `build/` 产物写入：

```text
work/character_packs/<package_id>/package/
├─ manifest.json
└─ roots/
   ├─ common/
   ├─ medium/
   ├─ android/
   └─ server/
```

随后运行 flow `status`，检查最后一行 JSON：

```powershell
$workspace = 'work/character_packs/<package_id>'
python mod-tools/wf_character_flow.py status --workspace $workspace
```

必须满足：

- `status.requirement_report.required_present = 37`；
- `status.requirement_report.missing_required = []`；
- `status.three_layer_claim_status.consistent = true`；
- `errors = []`；
- manifest、文件大小、SHA-256 和语义占用一致。

缺项时只补 `missing_required` 列出的真实逻辑路径。禁止复制相似角色、改名占位、复制排除目录，
也禁止只改 manifest/SHA 掩盖文件缺失。

## 9. 调整循环

调整分两类：

- 整体调整 `global`：如“整体更暗黑”“盔甲改成银白色”；
- 细节调整 `asset|text_field|voice_line|data_slot`：如“只让进化立绘表情更冷”、
  “这句台词慢一点”“三号词条换成防御向”。

网页中选择作用域和目标，输入增量指令，先查看“受影响项”，再确认。CLI/HTTP 的语义为：

```json
{
  "spec_id": "<spec_id>",
  "scope": "asset",
  "target": "evolved_master",
  "instruction": "表情更冷，但保持发色、瞳色、服装和配色不变"
}
```

依赖传播规则：

- 身份卡外观变化 → 母版、全部派生 UI、trim 定位和像素参考变 dirty；
- 只改基础或进化一侧 → 可只重做该侧及下游；
- 台词稿变化 → 只重做对应语音；
- 技能名/描述变化 → 对应文本表声明 dirty；
- 数据槽位变化 → 对应表行/DSL、QA、pack dirty。

旧产物保存在 `authoring/history/<asset>/<rev>/`，可并排比较并回退上一版。每次调整后都要重新
跑受影响阶段、S6 QA 和 S7 pack；哈希未变的其他项不应重做。

## 10. 预检

普通预检：

```powershell
python mod-tools/wf_character_flow.py preflight --workspace $workspace
```

production 的成功条件是：`release_ready=true`、`preflight.can_prepare=true`、
`preflight.conflicts=[]`、`preflight.writes_live=false`，且当前客户端基础版本能沿真实增量图
到达已验证尾版本。

runtime_test 按契约保留 `release_ready=false`，因此不能拿 production 的 `release_ready=true`
判据套它；应检查事务预检的 `can_prepare=true`、冲突为空、所有权凭证匹配，并保留用户当次授权。
当前 flow 会因为 runtime_test 不是 production-ready 而返回退出码 3；这不等于可以忽略其他门禁，
所有权、路径、manifest 或输入错误仍必须停止处理。

对 production，退出码 3 表示尚未达到发布条件；退出码 2 表示输入、结构或安全门禁错误。
按 `errors` 和 `next_command` 处理。live 表有变化时：

```powershell
python mod-tools/wf_character_flow.py rebase --workspace $workspace
python mod-tools/wf_character_flow.py preflight --workspace $workspace
```

自动化只能解析最后一行 JSON 的退出码、`ok`、`release_ready`、`errors` 和 `next_command`，
不能解析人类提示文本。

## 11. 发布与发放

### 11.1 单包所有权硬约束

`wf_character_flow.py` 维护的是单包所有权链。不能假设“每个新角色都能用一个新 package_id”。
production 包必须延续当前 active release 的所有权，并用与 active
`package_manifest_sha256` 绑定的已安装包作为凭证。

截至 2026-07-17 01:50 的实测状态：

- 当前 active 归属累计包 `seris_dragon_king`；
- 新角色 production 必须并入这个包：`package_id` 保持 `seris_dragon_king`，
  `character_id` 使用新角色 ID，并在累计包中新增该角色的表行和资产；
- 独立 `package_id` 不能直接发布；
- `runtime_test` 也经过同一所有权门，不能拿独立包绕过；
- 当前凭证包位于
  `.worktrees/seris-dual-form/work/sfix159/seris_dragon_king/package`。

上述路径是时点记录，不是永久常量。发布前必须重新核对 active 的 manifest 哈希仍与该凭证包
一致；不一致就停止并找回当前 active 对应的原已安装包。禁止从 live bytes 反推或伪造凭证。

如果 S0 已建立独立 package_id，先停在 plan/pack 阶段修正归属；不要继续尝试不同口令。

### 11.2 draft 的 runtime_test

manifest 必须明确：

```json
{
  "delivery_mode": "runtime_test",
  "release_ready": false,
  "user_authorized_direct_real_test": true
}
```

授权字段必须来自用户当次操作。绑定当前 active 凭证后重新预检：

```powershell
$workspace = 'work/character_packs/seris_dragon_king'
$installedPackage = '.worktrees/seris-dual-form/work/sfix159/seris_dragon_king/package'

python mod-tools/wf_character_flow.py preflight `
  --workspace $workspace `
  --installed-package-dir $installedPackage
```

用户决定真机测试、并满足 flow 实际停服门禁后，才可执行：

```powershell
python mod-tools/wf_character_flow.py publish `
  --workspace $workspace `
  --installed-package-dir $installedPackage `
  --confirm DIRECT_REAL_TEST
```

`DIRECT_REAL_TEST` 不能发布 production，`PUBLISH_CHARACTER_PACKAGE` 也不能把 runtime_test
包静默升级成 production。

### 11.3 complete 的 production 发布

1. 保存最终 QA 报告；
2. 运行 `status`；
3. 绑定已安装包运行 `preflight`；
4. live 漂移时 `rebase → preflight`；
5. 停止 CN 服务；
6. 用户输入精确口令发布；
7. 保存最后一行 JSON 和 `snapshot_dir`。

```powershell
$publishJson = python mod-tools/wf_character_flow.py publish `
  --workspace $workspace `
  --installed-package-dir $installedPackage `
  --confirm PUBLISH_CHARACTER_PACKAGE |
  Select-Object -Last 1 |
  ConvertFrom-Json

if (-not $publishJson.ok) { throw ($publishJson.errors -join '; ') }
$snapshotDir = $publishJson.snapshot_dir
if (-not $snapshotDir) { throw 'Publish output did not provide snapshot_dir' }
```

发布以原子替换 `active.json` 为提交点。不能在服务未停、摘要漂移、37/37 不足或 active 链不可
验证时继续。生成器界面不得自动代填 `PUBLISH_CHARACTER_PACKAGE`。

### 11.4 发放角色

发布成功后，通过现有 admin 邮件流程按新 `character_id` 发放。不在生成器内接卡池，也不修改
种子页或卡池组。领取后至少检查：角色一览、详情页、基础/进化立绘、技能/队长技/词条文案、
战斗模型、cut-in、像素序列和语音。

## 12. 回滚三件套

发布级回滚必须同时保存三件东西：

1. `snapshot_dir`：成功 publish 的最后一行 JSON 返回值；
2. `installed_package_dir`：发布时实际绑定并安装的原 package 目录，内容与 manifest 哈希必须保持；
3. 精确口令：`ROLLBACK_CHARACTER_PACKAGE`。

这三件缺任意一个都不要尝试回滚。尤其不要把后来继续修改过的 workspace `package/` 当成原
已安装包，也不要从当前 live 目录拼一个“差不多”的包。

停服后执行：

```powershell
python mod-tools/wf_character_flow.py rollback `
  --snapshot-dir $snapshotDir `
  --installed-package-dir $installedPackage `
  --confirm ROLLBACK_CHARACTER_PACKAGE
```

flow 回滚不是覆盖历史或降低 active 版本，而是发布一个更高版本的 `*-rollback` 反向增量。
成功前会校验 finalized marker、`snapshot.json` 摘要、active 链位置、installed package 的
canonical manifest hash、live 文件一致性、四根路径边界和停服状态。

旧快照没有 `snapshot_sha256` 或 finalized 摘要时，只能报告为旧格式。不得手工补字段、改哈希、
从 live bytes 猜测恢复内容。

发布前的内容回退不需要 flow rollback：

- 单资产/单文本：从 `authoring/history/<asset>/<rev>/` 回退修订；
- 整体规格：恢复 `spec.json` 的已知修订，再跑受影响阶段、QA 和 pack；
- 已经 publish：只能用上述回滚三件套走反向增量。

## 13. 常见问题

### 没有 Key，生成按钮不可用

检查用户级 `OPENAI_API_KEY` 是否存在，重新打开终端和 GUI。不要把 Key 粘到 spec 或日志。
如果只使用已有产物，仍可手动上传、QA 和 pack。

### GPT-SoVITS 端口在线，但 voice 失败

先确认当前版本是否真的提供 `local_http` provider；再核对权重配置、参考音频路径、参考文本、
语言代码和 `/tts` 实际合成。端口在线不等于推理可用。

### MP3 能播放，但 QA 拒绝

常见原因是 VBR、采样率/声道错误、Xing 头或转换未覆盖文件尾。用生成器给出的 ffmpeg 参数重新
转码，不要直接改扩展名或跳过 `mp3_encode`。

### 母版很好看，小头像却切坏

回到 UI 马赛克检查裁切焦点。优先调整共享焦点或母版构图后重新派生，不要逐张手修小图。

### 像素 repaint 某些帧漂移

只重生成失败帧，保持原姿势、剪影和透明边界；IoU 未达到 0.6 不能手动标绿。必要时退回
`recolor`，先完成角色整体流程。

### `missing_required` 不是空

只补报告列出的真实逻辑路径。`story`、`words` 等排除项不算缺失；不要复制相似角色资源占位，
不要只改 manifest。

### `STALE_RELEASE_BASE` 或 live drift

执行 `rebase → preflight`。rebase 只合并 manifest 声明拥有的行，不应覆盖未声明的 live WIP。

### 独立新包被所有权门挡住

这是安全门正常工作，不是换口令能解决。核对 active 所属累计包和凭证 manifest，把新角色并入
同一 package_id 后重新组包、status、preflight。

### runtime_test 也被挡住

runtime_test 不是绕过所有权链的后门。绑定 active 对应的已安装凭证包；仍不匹配就停止，保留
workspace、最后一行 JSON 和 active 证据，交由维护者判断。

### rollback 提示 installed package 不匹配

找回发布时冻结的原 package。不要改 manifest、不要从 live 反推、不要拿当前已继续编辑的
workspace 代替。
