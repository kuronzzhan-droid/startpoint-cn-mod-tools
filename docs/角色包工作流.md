# 新角色包工作流

本文是 CN 新角色从生成器输出到客户端发布的唯一操作入口。生成器只写 workspace，`wf_character_flow.py` 负责校验、重基、发布和反向增量回滚；不要让生成器直接写 live store、`.cdn` 或 `assets`。

## 1. 初始化 workspace

在项目根目录执行：

```powershell
$workspace = 'work/character_packs/seris_dragon_king'
python mod-tools/wf_character_flow.py init --template-id 111165 --character-id 129999 --code-name seris_dragon_king --package-id seris_dragon_king
python mod-tools/wf_character_flow.py status --workspace $workspace
```

`init` 只创建草稿，不代表可发布。固定目录如下：

```text
work/character_packs/<package_id>/
├─ workspace.json
├─ package/
│  ├─ manifest.json
│  └─ roots/{common,medium,android,server}/
└─ evidence/
   ├─ hash-cache.json
   └─ status.json
```

角色生成器必须把结果交给 `package/`：

- `roots/common|medium|android` 放客户端逻辑路径文件；
- `roots/server` 放服务器四层数据；
- `manifest.json` 声明每个文件的 SHA-256、大小、表键和语义占用；
- 不复制剧情、台词、登录语音或表情目录；
- `story`、`words`、`login`、`episode` 明确排除；动态 ally/battle/home 语音只作为建议项，不计入发布门槛。

## 2. 补齐并检查 37 项必需资产

反复执行：

```powershell
python mod-tools/wf_character_flow.py status --workspace $workspace
```

看最后一行 JSON：

- `status.requirement_report.required_present` 必须为 `37`；
- `status.requirement_report.missing_required` 必须为空；
- `status.three_layer_claim_status.consistent` 必须为 `true`；
- `errors` 必须为空。

状态计算使用相对路径哈希缓存，可中断后继续。它不会扫描 live store，也不会输出密钥或绝对 live 路径。

## 3. 只读预检和自动封存

```powershell
python mod-tools/wf_character_flow.py preflight --workspace $workspace
```

当资产已到 37/37、manifest 和三层声明完整时，`preflight` 会在 workspace 内自动写入最终 QA 数量和 `workspace_input_sha256`，随后执行事务预检。它不会写 live store 或 CDN。

成功条件：

- `release_ready=true`；
- `preflight.can_prepare=true`；
- `preflight.conflicts=[]`；
- `preflight.writes_live=false`；
- `requires_client_base` 可以通过真实增量图到达当前已验证尾版本。

退出码 `3` 表示预检尚未就绪；退出码 `2` 表示输入、结构或安全门禁错误。两者都不能继续发布。

## 4. 重基到当前 live 表

```powershell
python mod-tools/wf_character_flow.py rebase --workspace $workspace
python mod-tools/wf_character_flow.py preflight --workspace $workspace
```

`rebase` 只把 manifest 声明拥有的行合并到当前 live 全表，未声明行保持 live 值。生产模式会：

1. 在 workspace 的临时兄弟目录生成完整重基包；
2. 原子替换 `package/`；
3. 重新计算并封存 workspace 摘要；
4. 验证成功后删除临时备份。

任何替换或封存失败都会恢复原 `package/`，并保留重基输出供检查；不会写 live 或 CDN。

## 5. 停服并发布

先停止 CN 服务，再执行：

```powershell
$publishJson = python mod-tools/wf_character_flow.py publish --workspace $workspace --confirm PUBLISH_CHARACTER_PACKAGE | Select-Object -Last 1 | ConvertFrom-Json
if (-not $publishJson.ok) { throw ($publishJson.errors -join '; ') }
$snapshotDir = $publishJson.snapshot_dir
if (-not $snapshotDir) { throw 'Publish output did not provide snapshot_dir' }
```

生产发布只有在以下条件全部满足时才会写入：

- 精确口令 `PUBLISH_CHARACTER_PACKAGE`；
- `release_ready=true` 且摘要仍匹配；
- 37/37 必需资产；
- 当前服务已停止；
- active release 链和三根归档全部可验证；
- preflight 的 live 基线未漂移。

发布事务依次生成 staging、快照、三根增量包，最后以 `active.json` 替换作为提交点。输出 JSON 的 `snapshot_dir` 必须保存，后续回滚只接受这个目录。

升级同一角色包时，预检和发布增加已安装包绑定：

```powershell
$previousInstalledPackage = 'work/installed_character_packages/seris_dragon_king/1.0.0/package'
python mod-tools/wf_character_flow.py preflight --workspace $workspace --installed-package-dir $previousInstalledPackage
python mod-tools/wf_character_flow.py publish --workspace $workspace --installed-package-dir $previousInstalledPackage --confirm PUBLISH_CHARACTER_PACKAGE
```

### 5.1 多独立包共存（2026-07-17 起）

active 链支持多个 package_id 交错发布，所有权按包推导：每个 package_id 的所有者哈希是它在链上**最后一次发布**的 manifest 哈希（`*-rollback` 增量会把源包所有权回退到上一版）。规则：

- **全新独立包首发**：不传 `--installed-package-dir`，即使链尾归其他包所有也能通过所有权门。行级保护仍然生效——候选包声明的任何 outer/inner/semantic 键若已被 live 占用（即其他包的行），预检报 `occupied_without_hash_bound_prior_ownership` 冲突；未声明的 live 行若被改动，报 `unclaimed_change`。
- **升级已有包**：`--installed-package-dir` 必须指向**该包自己**上一次发布的 package 目录，哈希绑定到该包在链上的所有者条目（不再要求它是链尾）。
- **回滚限制不变**：只能回滚链上**最后一次**发布（不论哪个包）。若要回滚更早的包，需先回滚其后的发布。
- 正常包的 package_id 禁止以 `-rollback` 结尾（保留给回滚增量）。
- 发布前必须 `rebase`（或确认包表基于当前 live 构建），保证包内全表除自己声明的行外与 live 字节一致，否则其他包的行会以 `unclaimed_change` 拦下。

## 6. 发布反向增量回滚

回滚不是覆盖历史或把 `active.json` 倒退，而是发布一个更高版本的 `*-rollback` 增量。已安装包目录是强制绑定证据；缺失时零写入失败。

```powershell
$installedPackageDir = "$workspace/package"
python mod-tools/wf_character_flow.py rollback --snapshot-dir $snapshotDir --installed-package-dir $installedPackageDir --confirm ROLLBACK_CHARACTER_PACKAGE
```

回滚门禁包括：

- 精确口令 `ROLLBACK_CHARACTER_PACKAGE`；
- 新格式 finalized marker 和 `snapshot.json` 摘要匹配；
- snapshot 必须对应 active 链最后一次发布的前置状态；
- 已安装包的 canonical manifest hash 必须绑定当前 active release；
- 当前 live 文件必须仍与已安装包一致；
- snapshot 中所有 live 路径必须精确落在配置的四根目录；
- 服务必须停止。

旧快照若没有 `snapshot_sha256`，只报告为旧格式，不应手工补字段或从 live bytes 猜测重建。新发布会自动生成可回滚快照。

## 7. `runtime_test` 例外模式

`runtime_test` 只用于已经获得明确授权的既有实测包，不是正常新角色发布路径。它必须同时保留：

```json
{
  "delivery_mode": "runtime_test",
  "release_ready": false,
  "user_authorized_direct_real_test": true
}
```

发布口令只能是：

```powershell
python mod-tools/wf_character_flow.py publish --workspace <runtime-workspace> --confirm DIRECT_REAL_TEST
```

`DIRECT_REAL_TEST` 不能发布 production 包，`PUBLISH_CHARACTER_PACKAGE` 也不能把 runtime-test 包静默升级为 production。

## 8. 故障处理

- `missing_required` 非空：只补列出的逻辑路径，不复制排除目录。
- `manifest hash mismatch`：重新生成对应 root entry，不要只改 SHA 字段掩盖文件漂移。
- `cannot reach validated tail`：先修复客户端基础版本/增量图，不要伪造 `requires_client_base`。
- `SERVER_RESTART_REQUIRED`：停服后重新预检；不要绕过端口检查。
- `STALE_RELEASE_BASE` 或 live drift：重新执行 `rebase → preflight`。
- rollback 提示 installed package 缺失：找回发布时的原 package；不要从当前 live 反推所有权。
- marker、摘要或 active 链损坏：停止发布，保留 staging、snapshot 和 recovery journal 作为证据。

所有命令的最后一行都是稳定 JSON，可由 PowerShell `ConvertFrom-Json` 处理。自动化只应以 `ok`、`release_ready`、`errors`、`next_command` 和退出码作决定，不解析人类提示文本。
