# 新角色制作心得

以「苍海龙王·赛瑞斯(129999)/夏日女神·史黛拉(139999)」从零到上线的完整过程为底,
沉淀做**自制新角色**(非既有角色数值修改)时真正让人流血的教训。程序步骤见
《角色包工作流》,数据逆向细节见《角色数据逆向与修改指南》,本文只讲**规则与坑**。

赛瑞斯上线跨了 1.4.142→1.4.157 共十几个热修版本——大部分本可以在发布前避免。

---

## 一、三条总原则

### 1. 先例原则:每个 kind 必须在同一张表里有官方先例

客户端的描述生成器(AbilitySummarizer→getInstantContentString 链)是按官方数据长出来的,
**战斗层能跑 ≠ 角色页能渲染**。同一个 kind 在词条表安全、放进队长表就空引用崩溃(F1009),
三次实锤:720(强化弹射 override)、689(速度固定)、185(状态固有触发)进 leader 表全崩。

- 铁律:**队长表零先例 kind = 必崩,无例外。新效果进不了队长表就放词条表**
  (功能等价:技能发动与形态开始是同一瞬间)。
- 统计先例时要**过滤敌方行**:有些 kind 的"官方用例"全是敌人(如 43),对玩家角色不算先例。
- 语境错位型 kind 最致命:触发 198(协力·且主位)在队长语境查合击上下文=空引用;
  而冷门内容 kind 多数只是文本降级,不崩。方向也算语境:互协力元素覆盖里
  空串方向(协力位视角)放队长表=矛盾必崩,'(None)' 方向(主位视角)才能进队长表。

### 2. 解析器即法律:从反编译的生成解析器提取 schema

`pinball/master/generated/AbilityValues.as`、`LeaderAbilityValues.as` 的 `parseAtN`
函数体就是每列的合法值表:**函数体含 ClientError 且无空串分支 = 该列空串必炸**(C7101/C7050)。
手写 legality 规则永远挂一漏万,**解析器级扫描脚本比任何人工检查都全**——新行上线前必跑。

- 空串与 `(None)` 是两种东西:Option 列的 None 是**字面量 `(None)`**,空串可能被解析成
  `Some(null)`(如 time 列空串=0 段=词条不生效)或直接崩。
- 官方哨兵速查:前置 1-3 kind='0'、instant_precontent='(None)'、delay='0'、
  during 累积触发='(None)'、even_if_owner_dead='false'。
- 触发块参数布局按 kind 变化:puller 4/5/6/7/9 的属性组参数在 **puller+1 列**,不是 trigger+9。
  组格空串会渲染成"null角色",官方只用元素 token 或 '(None)'。

### 3. 崩溃排查范式:先拉日志,不要猜

- 客户端崩溃必发 `POST /crash`,服务端日志 `[CRASH]` 带堆栈,是第一手证据;
- 客户端本地 `latest.log` 的 errorMessage 带**完整 AS3 stackTrace + resourceVersion**,
  一发定位崩点;注意它每次重启会重置,要抢救(循环 grep stackTrace 转存);
- 错误码体系(CrashUtil 实锤):`F+数字`=未捕获 AS3 Error(F1009=空引用,找"引用了但
  不存在的东西"),`C`=ClientError(数据校验),`C8601`「指定的Key不存在」界面却显示
  "资源文件已损坏"——纯误导,先想"哪个 key 没注册";
- 拿到 parseAtNN 帧后,去反编译源找对应生成类,离线复刻其 case 路由扫全表,几分钟定位毒行。

## 二、崩溃图鉴(实测对照表)

| 现象/错误码 | 根因 | 修复 |
|---|---|---|
| U0000 | CSV 引号内含换行的行被逐行解析撕裂(character_text/speech) | 全文本 csv.reader 往返,字节级 0 差异校验 |
| C7050(开角色页) | 枚举列空串;或 element=6 写给可玩角色 | 补官方哨兵;可玩角色元素只能 0-5 |
| C7050(角色一览) | 词条毒行 + 列表关键字持久化(matchKeyword 全量遍历) | 先例扫描;注意"一打开列表就崩"假象 |
| H400 回登录 | 玛纳板节点 multiplied_id = 角色ID×2 拼板序,克隆未重映射→指回模板 | 三层树叶子重编号 + 服务端 mana_node.json 补条目(静态 import 须重启) |
| F1009(角色页) | 队长表零先例 kind(见先例原则) | 挪词条表/换官方先例组合 |
| C8601 | InvokeSkill 的 string_id 未注册 custom_ability_string 表 | 缺键自动注册;别信"资源损坏"字面 |
| C2308 | during 行错带 Initial 累积条件 | '0'→'(None)' |
| C7101 | content 0/486 族 Bool/multiply 参数列空串 | 按解析器 schema 补齐全部参数格 |
| U_f6da5a(技能发动) | 一条 CreateCondition 同时挂 3+ 条件且含独有状态(硬规则:含 ACUnique 时长度≤2) | DSL 拆两条命令,独有状态单独一条 |
| 领取演出即崩(硬 throw) | ★4+ 角色缺 special 槽或 special.timeline 缺 special_land/special_pose 标签 | 双形态角色龙形 timeline 战斗标签之外必须追加这两段 |
| 进战 FATAL | P-code 补丁 getlex 引用了新编译类 | 纯 P-code 只调**现有类现有 API**(真机二分实证) |

## 三、数据层心得

- **元素的落点**:c3=UI+球+克制+技能伤害元素;string token(Red/Blue/Yellow/Green/White/Black)
  只出现在己方队门槛单元格,翻 token 零误伤;**单改资料页 element 字段词条门槛不跟着变**,
  改属性必须整体转(c3+全套 token+element 数值列)。注意 **720 的元素参数是弹射元素序
  火水雷风光暗(3=雷,4=风)**,和 token 序不同,真机渲染实证过。
- **词条造伤三族**:kind 33/34/388 是增伤 buff 不是造伤;真造伤是 251/316/352 三族
  (元素编码在 kind 号里,time 列空=全体);词条层做不出"计为技能伤害"的直接伤害,
  要技能伤害只能 kind 629 InvokeSkill(string_id 必须注册,否则 C8601)。
- **主位限定**:只用 character 表 col1 unisonable=false;前置 kind 202 是零先例发明值,
  与 col1 同用会渲染双 Ⓜ 图标。
- **立绘三表契约**:full_shot PNG 官方是紧贴内容的裁剪图;
  `trimmed_image=(tx,ty,2000,2000)`、`character_image=(tx,ty,图宽,图高)`、
  `full_shot_image_attribute=(1000,1000,1,face_x,face_y)`;官方内容水平中心≈994、脚底≈1959。
  用整图时 tx/ty 要按内容 bbox 反推,抄模板值必然偏移。
- **玛纳板 ID 编码**:节点 multiplied_id = 角色ID×2 拼 板序×200+节点号,**客户端由节点 ID
  反推角色 ID**——克隆/新建必须重映射,这是 H400 的根源。

## 四、资产层心得

- **special 槽的本职是 ★4+ 获取演出**(join movie):`useSpecial()=rarity>3`,
  `gotoAndPlay("special_land"/"special_pose")` 缺标签硬 throw。双形态角色借 special 槽放
  第二形态时,**必须在同构战斗标签之外追加这两段**,否则邮件领取演出必崩。
- 战斗中队员只会播 9 个标准序列;第二形态与人形 9 标签**绝对 tick 同构**即可,追加段
  战斗永远播不到——这是"一个槽两用"能成立的机理。
- 表引用的每张图必须真实存在(赛瑞斯 wet 独有状态图标从未生成→战斗"数据不足"+下载卡死),
  发布门禁应有**表引用资产存在性检查**。
- 语音实装验证法:包内 mp3 与 character_speech 表 home/ally 行路径一一对应 + battle 命名
  惯例对上 = 已实装,不用逐条真机听。

## 五、客户端补丁心得(双形态类需求)

- **getlex 铁律**:P-code 补丁引用任何新编译类(marker/carrier)→ 进战必崩;
  纯 P-code 方法体替换 + 只调现有 API = 安全(stack-noop/existing-condition/host-only 三探针实证)。
- FFDec 整类回编不可靠(曾把运行时 guard 编译判假,C8601 随机崩),**数值哨兵 + P-code 级
  精准替换**是唯一稳定路径;哨兵契约(如 kind155+strength62500 被补丁钩子按值识别)进了
  数据就是 API,**数据侧永远不能改**,要改只能动补丁。
- 服务端/数据/客户端补丁**必须配套发布**:数据先行而补丁未装的客户端,轻则功能缺失
  (湿润倍率不生效),重则演出即崩。全员换包后再发角色。

## 六、发布链路心得

- 客户端主数据(②层 orderedmap)在 CDN 增量里,**不随 git 分支自动到手**——自托管者只
  clone 分支会缺这层(打完 15 轮/开商店才 C8601,1-14 轮静默,极难自查)。解法=把增量链
  打进 git 跟踪的 `assets/asset-patch/active/` 由服务端自动 serve。
- **版本号撞车**:客户端 ResVer 已等于 patch 版本号时服务端不会下发任何更新——增量链
  白费。根治=整合桥接包(目标表完整文件整替换,版本号必须**大于**客户端当前值)。
- `package_version` 是**字符串比较**(2.0.10 < 2.0.9 会判降级),版本规划直接用 2.1.0。
- 全表 payload 新增 outer key 必须同步进 manifest 的 `tables[].outer_keys`,否则
  unclaimed_change 冲突。
- 发布产物 zip 有 CI 卫生门禁(≤5MiB):角色包几十 MB 属常态,按序号拆分同一版本边即可
  (`pinball-<from>-<to>-<seq>-<suffix>.zip`,协议本就支持一版多包,客户端全量下载)。
- `wf_character_flow` 的 publish 前**必须先跑 preflight**(它刷新 release_ready)。

## 七、验证纪律

1. **克隆金丝雀先行**:发全新 ID 前,先零拷贝克隆一个官方角色(如 111165→119999)端到端
   走一遍:邮件发放→领取演出→角色一览(带关键字遍历)→详情页→玛纳板全学满→进战。
   金丝雀通过的链路,新角色只剩"数据本身"一个变量。
2. **用缺数据的环境验收**:作者自己的 CDN/客户端什么都有,测不出自托管者会遇到的缺层问题。
   干净 checkout / 低版本客户端才是有效验证环境。
3. **解析器级扫描 + 先例门禁进 CI**,而不是靠记忆;静态校验(数字/哨兵/布尔)挡不住
   "战斗层能跑但描述生成器没见过"的组合。
4. **真机兜底不可省**:渲染语义(720 元素序、"null角色"、双 Ⓜ)只有真机截图能定案。
   十几个热修版本里,凡是跳过真机直接推断的结论,后来几乎都被真机修正过。
